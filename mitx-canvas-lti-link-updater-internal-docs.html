Overview

The Canvas LTI Link Updater rewrites LTI (Learning Tools Interoperability) links in a Canvas course export so they point to a new edX course run. When an edX course is re-run or copied in Studio, block IDs are preserved but the course identifier (course-v1:ORG+COURSE+RUN) changes, so every Canvas LTI link to that course becomes stale. The tool parses both exports, matches links to blocks by block ID, rewrites matched URLs, and produces an updated .imscc plus an audit CSV.

There are three front ends over the same logic. Two implementations exist: a Python pipeline (src/, used by the CLI and Flask app) and a JavaScript port (docs/lti-core.js + docs/stream-core.js, used by the browser app). They are verified to produce byte-identical output.

Why the browser app streams

Processing a course entirely in a browser tab means everything is in memory. A naive load (loading the whole archive at once) holds the whole input zip plus the whole rebuilt output: measured ~930MB peak RSS for a 288MB input (~3×), which scales to ~2GB for a 600MB course and crashes tabs. The production app in docs/ instead streams: it decompresses the edX .tar.gz as a stream and keeps only the small hierarchy XML (never the media), reads the Canvas .imscc with random access (touching only XML, not media), and rewrites by stream-copying every entry to the output. Measured ~330MB peak RSS for ~530MB of input, flat regardless of media size. In Chrome/Edge the output streams to disk via the File System Access API; other browsers fall back to an in-memory Blob.

Local CLI Usage

  1. Put the Canvas export and the edX export in any folder. File names do not matter: the CLI identifies each file by content (ZIP containing imsmanifest.xml → Canvas; gzipped tar containing course.xml → edX).
  2. From the repository directory: 
    python cli.py /path/to/folder
  3. Outputs land in the same folder (override with -o /other/folder):
    • <name>_updated.imscc — the rewritten Canvas export
    • lti_audit_report.csv — all links, MISSING first

Re-runs are safe: files ending in _updated.imscc are ignored as inputs.

Unverified Rewrite Mode (--target)

When no edX export of the new course is available but the new course is known to be an exact copy (Studio re-runs and export/import copies preserve block IDs), all LTI links can be rewritten to a target course ID without block verification:

python cli.py /path/to/folder -t course-v1:MITx+ulmo_general_2+smoketest

Only a Canvas export needs to be present. Every link with a parseable course ID is rewritten to the target; links without one (e.g., the bare tool-configuration URL) are left unchanged and reported MISSING. The audit CSV carries an UNVERIFIED REWRITE warning. Prefer verified mode: it is the only way to catch links to blocks that were deleted or restructured. After an unverified rewrite, spot-check links post-import.

Result Statuses (Technical)

StatusMeaning
MATCHEDBlock ID from the Canvas LTI URL was found in the new course's block inventory (or rewritten in --target mode). URL updated in the output.
MISSINGBlock ID not found in the new course, or no course-v1:/block-v1: identifier could be parsed from the URL. URL left unchanged.
NEW_ONLYA linkable block (sequential or vertical) exists in the new course with no corresponding Canvas link. Informational; leaf components (problem/video/html) are deliberately excluded to keep the report readable.

Audit CSV columns: status, resource_title, old_lti_url, new_lti_url, block_id, block_type, edx_location, notes. Sorted MISSING → NEW_ONLY → MATCHED.

Technical Architecture

Five-phase pipeline, executed server-side in a single request (web) or sequentially (CLI):

  1. Phase 1, Parse Canvas IMSCC: CanvasLTIParser extracts the IMSCC ZIP, parses imsmanifest.xml for LTI External Tool resources, reads each basiclti_link.xml, and also scans assignment XML files and module_meta.xml for additional LTI references. Each URL is parsed for edX identifiers (org, course, run, block_type, block_id) via course-v1:/block-v1: regex patterns.
  2. Phase 2, Parse edX OLX: OLXParser extracts the .tar.gz, locates course.xml, reads org/course/run, and walks the hierarchy (chapter → sequential → vertical → components) to build the block inventory. The inventory includes chapters, sequentials, and verticals themselves — MITx LTI links most commonly target sequentials and verticals, not leaf components. macOS metadata files (._*, __MACOSX) are ignored.
  3. Phase 3, Map links: LTIMapper matches each Canvas link to a block by block ID. Matched links get new URLs by replacing org/course/run while preserving block type and ID. Unmatched links → MISSING. Unlinked sequentials/verticals → NEW_ONLY.
  4. Phase 4, Generate updated IMSCC: IMSCCUpdater copies the extracted directory, string-replaces old URLs in LTI XML files, sweeps all XML for remaining old course-v1/block-v1 references, and repackages as ZIP. Empty directories (e.g., wiki_content/) are preserved. The output is byte-identical to the input except for rewritten URLs.
  5. Phase 5, Audit CSV: generate_audit_csv() writes the report sorted MISSING-first.

URL Matching Logic

  • course-v1:{org}+{course}+{run} in the URL path
  • block-v1:{org}+{course}+{run}+type@{type}+block@{id} in the URL path
  • block-v1:... in query string parameters (usage_key, id, block_id)

Only org/course/run are replaced; block type and ID are preserved, since re-run/copied edX courses retain block IDs.

Security Notes

  • Tar path traversal: the OLX parser rejects archive members with absolute paths or ...
  • Download path traversal: /download/<filename> sanitizes with secure_filename() and verifies the resolved path stays inside the output folder. Requests like /download/../../etc/passwd (including URL-encoded variants) are rejected. Do not weaken this when modifying the route.
  • No outbound network requests; no credentials or database.

Hosting and Deployment

A. Browser app on GitHub Pages (primary)

The docs/ folder is a static site. To publish: repository Settings → Pages → Source = main branch, /docs folder. GitHub serves it at https://<org-or-user>.github.io/<repo>/. No build step, no server, no cost. It loads zip.js from a CDN (jsDelivr, pinned version) and uses the browser's native gzip decompression; everything else is local.

To test changes before pushing, serve the folder locally (cd docs && python -m http.server) and open the printed URL — opening index.html directly via file:// fails because browsers block module/CDN loading from the filesystem. There is no separate "deploy" step beyond pushing to main; Pages republishes automatically.

C. Hosted Flask app

Optionally hosted on PythonAnywhere. render.yaml contains a Render configuration for migration. Because it accepts uploads, large courses require a paid tier with raised upload limits and timeouts; otherwise direct users to the browser app or CLI.

PythonAnywhere

  • Runs as a WSGI web application.
  • Temp files in /tmp/lti_uploads and /tmp/lti_outputs, cleaned before each processing run.
  • Practical file size limit ≈ 70 MB due to disk constraints. (Larger courses: use the CLI.)

Render (Future)

  • Python 3.11.7 runtime, auto-generated SECRET_KEY, health check at /health, gunicorn with 300-second timeout.

Dependencies

From requirements.txt: Flask ≥ 3.0.0 (web framework), gunicorn ≥ 21.2.0 (production WSGI server), Werkzeug ≥ 3.0.0 (secure filename handling). Everything else is Python standard library (xml.etree.ElementTree, zipfile, tarfile, csv, urllib.parse); BeautifulSoup/lxml are not required. Runtime: Python 3.11.7 (runtime.txt).

pip install --upgrade -r requirements.txt

Common Failure Modes and Diagnostics

Disk Space Exhaustion

Both uploads are extracted to /tmp, and the IMSCC is copied to a working directory. A 70 MB IMSCC can need 200+ MB of temp space. The app cleans temp folders at the start of each request (cleanup_tmp_folders()) and after processing. If the server runs out of space, manually clear /tmp/lti_uploads, /tmp/lti_outputs, and any /tmp/canvas_lti_* or /tmp/olx_parse_* directories.

IMS Namespace Version Mismatch

Canvas IMSCC files can use IMS CC v1.1, v1.2, or v1.3 namespaces; the parser auto-detects from the root element. For a new namespace version, update NAMESPACES in canvas_lti_parser.py.

Non-Standard LTI URL Formats

Links without course-v1:/block-v1: patterns (non-edX tools, custom providers, the bare tool-config URL) are not rewritten — by design.

Wrong-File Pairings

The two most common user errors are exporting the new Canvas course instead of the old one, and exporting the old edX course instead of the new one. Symptoms: "same course version" warnings, everything MISSING, or a no-op output where old and new URLs are identical. The web UI's confirmation step surfaces the detected course IDs before processing precisely to catch this; when triaging, always check the audit CSV's old/new course IDs first.

Gunicorn Timeout

300-second timeout in Procfile/render.yaml. Very large courses may approach it; processing scales with XML file count in both exports.

Logging

INFO-level structured logs per pipeline phase; warnings for individual file failures. Check the PythonAnywhere error log or server console.

How to Restart / Redeploy

PythonAnywhere

  1. Log in to PythonAnywhere.
  2. Web tab → "Reload" next to the app.
  3. Verify /health returns {"status": "healthy"}.

Deploying Updates

  1. Upload updated code (Files tab or git pull).
  2. If dependencies changed: pip install -r requirements.txt in a console.
  3. Reload from the Web tab.

Local Development

pip install -r requirements.txt
python app.py

Flask dev server on http://localhost:5000.
