Overview

The Canvas LTI Link Updater rewrites LTI (Learning Tools Interoperability) links in a Canvas course export so they point to a new edX course run. When an edX course is re-run or copied in Studio, block IDs are preserved but the course identifier (course-v1:ORG+COURSE+RUN) changes, so every Canvas LTI link to that course becomes stale. The tool parses both exports, matches links to blocks by block ID, rewrites matched URLs, and produces an updated .imscc plus an audit CSV.

There are two front ends over the same logic. Two implementations exist: a Python pipeline (src/, used by the CLI) and a JavaScript port (lti-core.js + stream-core.js, used by the browser app). A CI parity check verifies they produce identical output (see Parity Check).

(An earlier hosted Flask app was removed — it duplicated the pipeline and required paid hosting for large uploads; the browser app covers its use case with no server at all.)

Why the browser app streams

Processing a course entirely in a browser tab means everything is in memory. A naive load (loading the whole archive at once) holds the whole input zip plus the whole rebuilt output: measured ~930MB peak RSS for a 288MB input (~3×), which scales to ~2GB for a 600MB course and crashes tabs. The production app instead streams: it decompresses the edX .tar.gz as a stream and keeps only the small hierarchy XML (never the media), reads the Canvas .imscc with random access (touching only XML, not media), and rewrites by stream-copying every entry to the output. Measured ~330MB peak RSS for ~530MB of input, flat regardless of media size. In Chrome/Edge the output streams to disk via the File System Access API; other browsers fall back to an in-memory Blob.

Browser requirements

The app requires DecompressionStream (Chrome/Edge 80+, Firefox 113+, Safari 16.4+). Unsupported browsers get a friendly "please use a current browser" message on load instead of a mid-run failure. Streaming save-to-disk requires Chrome/Edge (showSaveFilePicker); other browsers build the output in memory and download it, which is fine for all but the very largest courses.

Local CLI Usage

  1. Put the Canvas export and the edX export in any folder. File names do not matter: the CLI identifies each file by content (ZIP containing imsmanifest.xml → Canvas; gzipped tar containing course.xml → edX).
  2. From the repository directory: 
    python cli.py /path/to/folder
  3. Outputs land in the same folder (override with -o /other/folder):
    • <name>_updated.imscc — the rewritten Canvas export
    • lti_audit_report.csv — all links, MISSING first

Re-runs are safe: files ending in _updated.imscc are ignored as inputs. The CLI is pure Python standard library — no pip install needed.

Unverified Rewrite Mode (--target / Advanced)

When no edX export of the new course is available but the new course is known to be an exact copy (Studio re-runs and export/import copies preserve block IDs), all LTI links can be rewritten to a target course ID without block verification. Available in both front ends:

CLI:

python cli.py /path/to/folder -t course-v1:MITx+ulmo_general_2+smoketest

Browser app: expand "Advanced: don't have the new MITx export?" on the upload screen, tick the checkbox, and enter the target course ID. Only the Canvas export is then required.

Every link with a parseable course ID is rewritten to the target; links without one (e.g., the bare tool-configuration URL) are left unchanged and reported MISSING. The audit CSV carries an UNVERIFIED REWRITE warning. Prefer verified mode: it is the only way to catch links to blocks that were deleted or restructured. After an unverified rewrite, spot-check links post-import. Both front ends produce identical output in this mode (covered by the parity check).

Result Statuses (Technical)

StatusMeaning
MATCHEDBlock ID from the Canvas LTI URL was found in the new course's block inventory (or rewritten in --target mode). URL updated in the output.
MISSINGBlock ID not found in the new course, or no course-v1:/block-v1: identifier could be parsed from the URL. URL left unchanged.
NEW_ONLYA linkable block (sequential or vertical) exists in the new course with no corresponding Canvas link. Informational; leaf components (problem/video/html) are deliberately excluded to keep the report readable.

Audit CSV columns: status, resource_title, old_lti_url, new_lti_url, block_id, block_type, edx_location, notes. Sorted MISSING → NEW_ONLY → MATCHED, then by title, block ID, and old URL (fully deterministic).

Technical Architecture

Five-phase pipeline, executed in the browser (streaming) or by the CLI:

  1. Phase 1, Parse Canvas IMSCC: CanvasLTIParser extracts the IMSCC ZIP, parses imsmanifest.xml for LTI External Tool resources, reads each basiclti_link.xml, and also scans assignment XML files and module_meta.xml for additional LTI references. Each URL is parsed for edX identifiers (org, course, run, block_type, block_id) via course-v1:/block-v1: regex patterns.
  2. Phase 2, Parse edX OLX: OLXParser extracts the .tar.gz, locates course.xml, reads org/course/run, and walks the hierarchy (chapter → sequential → vertical → components) to build the block inventory. The inventory includes chapters, sequentials, and verticals themselves — MITx LTI links most commonly target sequentials and verticals, not leaf components — and each block carries its chapter/sequential/vertical location context. macOS metadata files (._*, __MACOSX) are ignored.
  3. Phase 3, Map links: LTIMapper matches each Canvas link to a block by block ID. Matched links get new URLs by replacing org/course/run while preserving block type and ID. Unmatched links → MISSING. Unlinked sequentials/verticals → NEW_ONLY.
  4. Phase 4, Generate updated IMSCC: IMSCCUpdater copies the extracted directory, string-replaces old URLs in LTI XML files, sweeps all XML for remaining old course-v1/block-v1 references, and repackages as ZIP. Empty directories (e.g., wiki_content/) are preserved. The output is identical to the input except for rewritten URLs.
  5. Phase 5, Audit CSV: generate_audit_csv() writes the report sorted MISSING-first.

URL Matching Logic

  • course-v1:{org}+{course}+{run} in the URL path
  • block-v1:{org}+{course}+{run}+type@{type}+block@{id} in the URL path
  • block-v1:... in query string parameters (usage_key, id, block_id)

Only org/course/run are replaced; block type and ID are preserved, since re-run/copied edX courses retain block IDs.

Security Notes

  • Tar path traversal: the OLX parser rejects archive members with absolute paths or ...
  • No server: the browser app processes files entirely client-side; nothing is uploaded. No outbound network requests; no credentials or database.

Hosting and Deployment (GitHub Pages)

The browser app lives at the repository root and is served by GitHub Pages. To publish: repository Settings → Pages → Deploy from a branch → main branch, / (root) folder. GitHub serves it at https://kalebabebe.github.io/mitx-canvas-lti-link-updater/. No build step, no server, no cost. All assets are local to the repo (zip.js is vendored in vendor/, pinned at 2.7.45), so there is no CDN dependency and the app works on restricted networks. A .nojekyll file disables Jekyll processing.

To test changes before pushing, serve the repo locally (python -m http.server from the repo root) and open the printed URL — opening index.html directly via file:// fails because browsers block script loading from the filesystem. There is no separate "deploy" step beyond pushing to main; Pages republishes automatically (allow a minute or two).

To upgrade zip.js: replace vendor/zip-full.min.js with the new dist/zip-full.min.js from the @zip.js/zip.js npm package, update the pinned version in package.json, and run the parity check.

Two Implementations and the Parity Check

Any change to matching/rewriting logic must be made in both pipelines — Python (src/) and JS (lti-core.js / stream-core.js) — including user-facing warning text, sort orders, and CSV formatting. The guard rail is tests/parity_check.js, which runs both pipelines on the fixtures in tests/ in both modes (verified and --target) and asserts identical output: byte-identical audit CSVs, identical file sets and per-file contents in the updated .imscc, and identical empty-directory preservation. (Raw zip bytes differ between Python's zipfile and zip.js — compression metadata — so the comparison is per-entry.)

It runs in GitHub Actions (.github/workflows/parity.yml) on every push and pull request. Locally:

npm install
npm test

Requires Node 18+ and Python 3.9+. If it fails after your change, one pipeline drifted — fix the other to match before merging.

Dependencies

  • Python (CLI): standard library only (xml.etree.ElementTree, zipfile, tarfile, csv, urllib.parse). Python 3.9+. No pip install.
  • Browser app: vendored zip.js (vendor/zip-full.min.js, BSD-3 license included) plus native browser APIs (DecompressionStream, File System Access where available). No build step, no CDN.
  • Dev/test only: @zip.js/zip.js and @xmldom/xmldom from npm, used by the parity check to run the JS pipeline headlessly in Node.

Common Failure Modes and Diagnostics

Unsupported Browser

Browsers without DecompressionStream (notably Safari < 16.4) get a clear message on page load. Direct the user to Chrome, Edge, Firefox, or a newer Safari.

IMS Namespace Version Mismatch

Canvas IMSCC files can use IMS CC v1.1, v1.2, or v1.3 namespaces; the parser auto-detects from the root element. For a new namespace version, update NAMESPACES in canvas_lti_parser.py (and mirror in lti-core.js, which matches by local name and is namespace-agnostic).

Non-Standard LTI URL Formats

Links without course-v1:/block-v1: patterns (non-edX tools, custom providers, the bare tool-config URL) are not rewritten — by design.

Wrong-File Pairings

The two most common user errors are exporting the new Canvas course instead of the old one, and exporting the old edX course instead of the new one. Symptoms: "same course version" warnings, everything MISSING, or a no-op output where old and new URLs are identical. The confirmation step surfaces the detected course IDs before processing precisely to catch this; when triaging, always check the audit CSV's old/new course IDs first.

Very Large Courses

The browser app streams and shows per-file progress during the rebuild; a 600MB course takes on the order of a minute. If a user's machine still struggles, the CLI processes the same files locally with no browser overhead.
